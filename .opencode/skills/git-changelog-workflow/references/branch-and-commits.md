# Branches and commits — repository conventions

## Branch naming

Pattern: `{type}/{description-in-kebab-case}`

| Prefix | Usage | Example |
|--------|-------|---------|
| `feat/` | New feature | `feat/cms-page-block-editing` |
| `fix/` | Bug fix | `fix/restore-blog-admin-routes` |
| `docs/` | Documentation only | `docs/agents-md` |
| `refactor/` | Refactoring without functional change | `refactor/translation-architecture` |
| `test/` | Test addition or fix | `test/admin-crud-coverage` |
| `chore/` | Maintenance, deps, CI | `chore/update-phpunit` |
| `skill/` | Agent Skills (`.opencode/skills/`) | `skill/git-changelog-workflow` |

### Rules

- Lowercase, single hyphens, no spaces
- No consecutive `--`
- Short description (3–5 words)
- One branch = one coherent topic for review and revert

## Commit messages (Conventional Commits)

```
type(scope): short summary in imperative

Optional body: context and motivation.
```

### Common types

| Type | When |
|------|------|
| `feat` | New feature |
| `fix` | Bug fix |
| `docs` | Documentation, CHANGELOG, skills |
| `refactor` | Restructuring without behavioral change |
| `test` | Tests |
| `chore` | Tooling, config, dependencies |
| `style` | Formatting without logic change |

### Scope (optional)

Examples for this project: `admin`, `cms`, `blog`, `auth`, `changelog`, `skill`,
`seo`, `translations`, `import-export`, `extension-manager`.

### Body conventions

- **`fix:` commits** MUST include `Regression test: #<issue_number>` in the body
- **`feat:` commits** MUST include `Feature test: <description>` in the body

### Examples from history

```
fix(blog): strip duplicate H1 from demo post bodies

Regression test: #23
```

```
feat(admin): add status filter on post list

Feature test: status_filter_shows_only_published_posts
```

```
docs(skill): add OWASP critical audit skill
docs(changelog): document sprints 3 through 7
feat(cms): add per-translation block editing
fix(seo): generate canonical URL from translation slug
```

## Atomic commits — criteria

A commit is **atomic** when:

1. It compiles / relevant tests pass
2. It represents **one** intent (one fix, one feature, one doc)
3. The message describes that intent without "and also"
4. A `git revert` cleanly undoes one logical change

### Typical breakdown

**Bad** — single commit:

```
feat(admin): filters, CSV export and slug fix
```

**Good** — three commits:

```
feat(admin): add status filter on post list
feat(admin): export posts as CSV
fix(blog): reject duplicate slugs on create
```

## Files to exclude from commits

- `.env` and any secret files
- `vendor/`, `node_modules/` (managed by lockfiles)
- `storage/app/private/`, `storage/framework/cache/`, sessions, compiled views
- `resources/dist/` (built assets committed separately during releases)
- `testbench-workbench/` (generated by Testbench)
- Local untracked artifacts listed by `git status`

## Branch → MR workflow

```
main ──► feat/my-feature ──► atomic commits ──► push ──► MR into main
```

Do not accumulate directly on `main`. Rebase or merge `main` into the feature branch if it is old, after user agreement.
